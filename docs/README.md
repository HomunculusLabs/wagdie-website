# Documentation Index and Policy

`docs/` is the canonical home for durable WAGDIE Simplified documentation. Top-level files such as `README.md`, `SETUP.md`, `ARCHITECTURE.md`, `DESIGN_SYSTEM.md`, and `STORYBOOK-QUICKSTART.md` are entry points or compatibility pointers, not places to keep duplicated procedures.

## Target information architecture

```text
docs/
  README.md                       Docs map, lifecycle policy, source-of-truth rules
  onboarding/                     Contributor setup and troubleshooting
  architecture/                   Current system seams and subsystem overviews
  development/                    Component, Storybook, design-system, and testing guidance
  operations/                     Repeatable operational procedures and validation runbooks
  reference/                      Maintained indexes such as route/API areas
  plans/                          Dated planning records
  reviews/                        Dated critiques/reviews
  investigations/                 Dated debugging and research records
  runbooks/                       Repeatable operational procedures with validation dates
  archive/                        Historical or immutable records and data exports
```

The rewrite plan that established this IA is `docs/plans/documentation-rewrite-2026-05-11.md`.

## Lifecycle definitions

- **Evergreen candidate**: Contains durable facts that may belong in onboarding, architecture, development, operations, or reference docs. Promote only after checking current code and canonical sources.
- **Runbook**: A repeatable operational procedure. Runbooks must say what they validate, where command/source truth lives, and when the procedure was last checked.
- **Historical**: A dated plan, report, investigation, critique, audit, or completion record. Historical docs may inform current docs but must not be linked as current behavior without a dated caveat.
- **Archive**: Historical material kept for traceability, compatibility, or data-retention reasons. Archive content is immutable by default except for index updates.
- **Superseded**: A document whose guidance has been replaced or invalidated. Keep it only as history, with a link to the replacement when one exists.

Classification rule: if a document describes a one-time change and that change has merged, failed, or been superseded, classify it as historical. If it describes a repeatable procedure, classify it as a runbook. If it describes current architecture or contributor workflow, verify it against source before promoting durable facts into evergreen docs.

## Source-of-truth rules

Docs summarize; code and configuration decide.

- Runtime: `.nvmrc` and `package.json` `engines.node` are canonical. This repo requires Node `23.3.0`.
- Commands: `package.json` `scripts` is the full command inventory.
- Package manager / runner: examples in docs should be Bun-first.
- Schema: `supabase/migrations/` is schema truth. Avoid hardcoded table-count checklists in prose.
- Routes and APIs: `app/api/*`, route handlers, and their tests are route-contract truth.
- Storybook: `.storybook/main.ts` and `.storybook/preview.tsx` are configuration truth. Do not hardcode story counts unless generated by an explicit command.
- Subsystems: cite the relevant implementation files instead of copying large code paths into docs.

When a doc claim conflicts with source, fix the doc or mark it historical/superseded.

## Bun-first command policy

Use Bun for install and package-script examples:

```bash
bun install
bun run dev
bun run build
bun run test
bun run storybook
```

Keep command examples minimal. For the complete and current list, link to `package.json` `scripts`. If a package script internally invokes `npx`, document the package script (`bun run <script>`) rather than asking contributors to run the internal command directly.

## Compatibility-pointer format

When an old entry point is replaced by a durable doc, replace its body with this short pointer shape:

```md
# <Old Document Title>

> Compatibility pointer: this content moved to `<replacement-path>`.

Use [`<replacement-path>`](<replacement-path>) for current guidance.
```

Rules:

- Keep one short banner and one link.
- Do not keep duplicate procedural content in the pointer file.
- If the replacement does not exist yet, leave the old file unchanged until the new page is ready or mark it as historical in the relevant index.
